Design Rationale

Design Decisions

We have used a hybrid architecture approach to ensure we provide a resilient, scalable and highly available solution to our customers. We will now go over specifics of our design choices and explain the rationale behind them.

Object Oriented Design

Object-oriented architectural styles are focused on the data. When modelling a system in object-oriented design, we begin by looking at the different kinds of data handled by the system to see how the system can be broken down into abstract data types. UserFeedback system is a perfect fit since it is easy to model the entities in form OOD.

Advantages of Object-Oriented Architecture

  • Modularity for easier troubleshooting

    OOP methods make code more maintainable. Identifying the source of errors becomes easier because objects are self-contained (encapsulation). The principles of good OOP design contribute to an application’s maintainability.

  • Reuse of code through inheritance

    Because objects contain both data and functions that act on data, objects can be thought of as self-contained “boxes” (encapsulation). This feature makes it easy to reuse code while enhancing systems.

  • Scalable

    OO applications are more scalable then their structured programming roots. As an object’s interface provides a roadmap for reusing the object in new software, it also provides all the information needed to replace the object without affecting other code. This makes it easy to replace old and aging code with faster algorithms and newer technology.

Repository Based

Whenever components need to share and reuse data between them, there are two common issues that emerge:

  • The state of a component is only temporary while the component is running. This means that any objects created and any amount of data processing will be lost when the component stops running
  • Components may not need all the data that is stored in the files. There needs to be a way to communicate the state of the data between multiple components

To solve both of the issues, a data-centric software architecture is used. This architecture can be achieved by integrating a method of shared data storage, such as a database, into the overall system design

  • Central data is the component used to store and serve data across all components that connect to it.
  • Data accessors are the components that connect to the central data component. The data accessors make queries and transactions against the information stored in the database

Since in this system we have a number of components working with the same data simultaneously, we have chosen to use the repository pattern and use a relational database to store all the information persistently, while the DAO layers in each component will assume the job to talk to the database

Databases ensure several data qualities, which are important to data-centric architectures. These are:

  • Data integrity. A database ensures that data is accurate and consistent over its lifespan.
  • Data persistence. A database ensures that data will live on after a process has been terminated—databases can save data from any number of components.

Layered Architecture

Layering allows for applying “separation of concerns” into each of the layers. This layering provides additional benefits as well such as scalability, resilience, maintainability, all of which are important non-functional requirements for our system, hence we have chosen to use this architectural pattern. Because of our utilization of this pattern, our business logic layer is entirely contained within the APIs.

Advantages of Layered Architecture

  • In software, a layer is a collection of components that work together towards a common purpose
  • The key characteristic of a layered architecture is that the components in a layer only interact with components in their own layer or adjacent layers - Principle of least knowledge
  • The interfaces provided by the components of a layer should be well defined and driven by the needs of the system
  • It supports the separation of concerns, because each layer is a set of components with similar responsibility or purpose
  • It is modular and loosely coupled, as each layer only communications with one or two other layers, allowing for different implementations to be easily swapped
  • It can be adapted so that layering is not always strict, which helps manage design complexity or provide a starting point for structuring the system

Microservices Architecture

We have used microservices in favor of a monolithic architecture. This helps in maintainability of the software as very specific component or service needs to be modified and redeployed whenever there is a change.

Advantages of microservices architecture

  • Easier to Build and Maintain Apps

    The key principle of microservices is simplicity. Managing the code becomes less painful because each microservice is, in fact, a separate chunk of code. Services can be implemented using different programming languages, databases and software environments. This allows each service to be deployed, rebuilt, re-deployed and managed independently.

  • Improved Productivity and Speed

    The microservices architecture tackles the problem of productivity and speed by decomposing applications into manageable services that are faster to develop. Different teams can be working on different components simultaneously without waiting on each other.

  • Scalability

    The microservice architecture allows decoupled services written in different programming languages to peacefully coexist with other fragments. With microservices, individual services can scale separately from one another.
